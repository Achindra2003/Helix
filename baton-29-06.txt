  Sense 1: "Open a teammate's chat, read it, and continue it with its context" ‚Üí
  ‚úÖ YES, this is real

  This is the actual shared-workspace identity, and it works end-to-end:

  - The sidebar lists all shared conversations in the workspace (GET 
  /conversations), so you can click into a teammate's thread and read it.
  - When you send a message, the route (router.py:176) only takes a branch_id ‚ÄĒ
  it does not care who you are. Your turn is appended, then build_messages
  (context.py:42) renders the entire branch history ‚ÄĒ every author's turns ‚ÄĒ 
  into the model's context, each user turn tagged [author] (context.py:63-64).
  - The system prompt itself (context.py:24) tells the model it's "the single
  shared AI assistant inside a team's collaborative workspace" where "different
  teammates may have written the user messages."

  So if you open Alice's shared thread and ask a follow-up, the AI answers using
  Alice's full context ‚ÄĒ and she gets yours when she continues. That's the
  BLUEHERON test that was proven live. This is genuinely the "shared LLM 
  workspace."

  Sense 2: "Pull a separate chat's context into a different chat of mine" ‚Üí ‚ĚĆ 
  Not as a standalone feature

  There is no "import context from conversation X into unrelated conversation Y"
  mechanism. build_messages only ever sees one branch's lineage
  (get_history(branch_id)); there's no cross-conversation merge, no @reference 
  another thread. If conversation A and conversation B are separate trees, B
  never sees A's context.

  The bridge between them: Fork (this is probably what you actually want)

  Fork is the closest thing to "take their chat's context and make it mine to
  run with":

  - POST /{conversation_id}/fork creates a new branch off their thread that
  inherits the entire ancestor spine (all their context up to the fork point),
  then diverges in isolation ‚ÄĒ your continued messages don't touch their
  original.
  - So the real workflow is: open a teammate's conversation ‚Üí Fork it ‚Üí now you 
  have your own line that carries all their context, and you continue privately
  from there.

  The nuance: a fork creates a branch inside the same conversation tree, not a
  brand-new top-level conversation owned by you. It's your divergent line within
  the shared thread, not a copy parachuted into an unrelated chat.

  ---
  Bottom line: Reading a teammate's chat and continuing it with full context ‚ÄĒ
  works today. Forking their chat to inherit its context and diverge on your own
  ‚ÄĒ works today. Merging one chat's context into a totally separate chat ‚ÄĒ that
  doesn't exist (and isn't on the roadmap as written).

  Is sense 2 ‚ÄĒ pulling context across unrelated conversations ‚ÄĒ something you
  actually want for the demo? If so, that's a real feature I could design (e.g.
  a "reference another thread" / context-import), but it'd be new work, not
  something that's already there.

‚ĚĮ how does it work?                                                             

‚óŹ Here's the full mechanism, traced end to end through the code I wrote.

  The core idea

  A reference is a live pointer, not a copy. When you link thread A into thread
  B, nothing from A is duplicated into B. Instead, B stores "I read from A," and
  every time you send a turn in B, the server re-fetches A's current context 
  and hands it to the model as background ‚ÄĒ then throws that assembly away. Next
  turn, it re-fetches again. That's why it stays in sync: there's no stored
  snapshot to go stale.

  1. Storing the link

  When you click Ôľč link context, the frontend calls POST 
  /conversations/{B}/references with {referenced_conversation_id: A}. The server
  validates (A exists, same workspace, A is shared, A‚Č†B) and writes one row to
  the new conversation_references table:

  conversation_id = B   |   referenced_conversation_id = A

  That's the entire persistent footprint. A directed edge B‚ÜíA. No messages
  moved.

  2. What happens when you send a turn in B

  Trace POST /conversations/{branchB}/messages (router.py):

  send_message(branchB)
    ‚Üí branch = get_branch(branchB)              # branch knows its
  conversation_id (B)
    ‚Üí references = _resolve_reference_blocks(B) # ‚Üź the live part
    ‚Üí ChatProducer(provider, references)
    ‚Üí engine.send(...)                          # persists your user node, runs
  producer

  _resolve_reference_blocks(B) is where "live" happens. It:
  1. reads the link rows for B ‚Üí [A]
  2. for each, fetches A's conversation, then calls
  get_history(A.default_branch_id) ‚ÄĒ A's current messages, right now
  3. wraps each as a ReferenceBlock(title, history)

  Because this runs on every turn, if your teammate added to A five seconds ago,
  this turn already sees it.

  3. How the model actually receives it

  ChatProducer.run calls build_messages(history, references=...) (context.py).
  The assembled prompt looks like:

  [system]  You are the single shared AI assistant... (the normal frame)
  [system]  The user has linked other conversations as background context.
            Use them as supporting reference, but they are NOT part of this
            thread's own turn-by-turn history...
            --- Referenced conversation: "Database decisions" ---
            alice (user): our primary datastore is CockroachDB
            assistant: noted
  [user]    [alice] Which datastore did the team pick?     ‚Üź B's own turns
  [assistant] ...

  The key design choice: A's content rides in a second system message, never as 
  user/assistant turns of B. So the model can ground on A, but A's messages
  never masquerade as B's history, never pollute B's lineage, and never get
  persisted into B. The test
  test_linked_reference_context_grounds_reply_without_polluting_lineage asserts
  exactly this ‚ÄĒ A's text shows up in the system frame, while B's non_system
  messages are still just B's own.

  That's the whole difference between this and a fork: a fork inherits an
  ancestor spine inside one conversation tree by walking parent_id (the nodes
  literally become part of the branch's history). A reference injects a separate
  tree's context as side material at prompt-assembly time and discards it after
  the call.

  4. Why "PostgreSQL ‚Üí CockroachDB" in the proof

  - Unlinked: references was empty, build_messages produced only B's frame + B's
  question. The model had zero grounding, so it pattern-matched "primary
  datastore" ‚Üí guessed PostgreSQL.
  - Linked: _resolve_reference_blocks pulled A's history, the second system
  frame carried "CockroachDB," and the model read it straight out.

  The seams that made it clean

  The whole feature slots into existing boundaries without disturbing them:
  - get_history (already the fork read-path) is reused verbatim to read the
  referenced thread.
  - build_messages got one optional parameter; the deep-reasoning producer is
  untouched.
  - engine.send never even knows references exist ‚ÄĒ they're resolved in the
  route and baked into the ChatProducer before the engine runs.

  One consequence worth knowing: references are resolved per turn but not 
  recursive ‚ÄĒ _resolve_reference_blocks reads A's messages, not A's links, so if
  A references C, your B does not transitively pull C. That's deliberate ‚ÄĒ it
  bounds the prompt size and makes loops impossible.
