      827 +## 8. The Branch–Node Tree: Physical Model for O(1) Fork      
      828 +                                                              
      829 +The central data-structure decision. A conversation's history 
          +is a **tree of immutable                                      
      830 +nodes**; a **branch** is a named pointer into that tree (a `fo
          +rk_node_id` it diverged at                                    
      831 +and a `head_node_id` tip). Forking therefore inserts **one** `
          +branches` row and copies                                      
      832 +**nothing** — it is constant-time no matter how long the conve
          +rsation is. The cost moves                                    
      833 +to the *read* path, which reconstructs a branch's history by w
          +alking `parent_id` from the                                   
      834 +head back up through the spine, transparently crossing into th
          +e parent branch's nodes —                                     
      835 +exactly Git's model of structural sharing. (The traversal algo
          +rithm itself is specified in                                  
      836 +the Algorithm-Design part of this document.)                  
      837 +                                                              
      838 +```                                                           
      839 +   main:     n1 ── n2 ── n3 ── n4(head)                       
      840 +                    │                                         
      841 +                    └── fork here → branch "chunk-v2"         
      842 +   chunk-v2:        (fork_node_id = n2) ── n5 ── n6(head)     
      843 +                                                              
      844 +   reading chunk-v2's history  =  walk n6 → n5 → n2 → n1   (ro
          +ot → head)                                                    
      845 +```                                                           
      846 +                                                              
      847 +For a **Deep-Reasoning** fork, the server additionally copies 
          +the engine's LangGraph

          + the recursive state                                          
      849 +continues independently; plain chat branches leave `engine_thr
          +ead_id` null.                                                 
      850 +                                                              
      851 +## 9. Multi-Tenant Isolation & Row-Level Security             
      852 +                                                              
      853 +Isolation is defence-in-depth, applied at two layers:         
      854 +                                                              
      855 +1. **Application layer.** Every query is filtered by the calle
          +r's `workspace_id`, resolved                                  
      856 +   from their membership; a request touching a workspace the c
          +aller is not a member of is                                   
      857 +   rejected before any row is read.                           
      858 +2. **Database layer (production Postgres).** Row-Level Securit
          +y policies on every                                           
      859 +   tenant-scoped table constrain access to the active workspac
          +e, set per request via a                                      
      860 +   session variable:                                          
      861 +                                                              
      862 +   ```sql                                                     
      863 +   ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;       
      864 +   CREATE POLICY tenant_isolation ON conversations            
      865 +     USING (workspace_id = current_setting('helix.workspace_id
          +')::uuid);                                                    
      866 +   -- the request handler runs:  SET LOCAL helix.workspace_id 
          += '<wid>';                                                    
      867 +   ```                                                        
      868 +                                                              
      869 +   Even a coding mistake in a query cannot leak across tenants
          +, because the database                                        
      870 +   itself filters the rows. (SQLite dev relies on the applicat
          +ion layer only; RLS is a                                      
      871 +   production-Postgres feature.)                              
      872 +                                                              
      873 +## 10. Visibility & Access at the Data Layer                  
      874 +                                                              
      875 +Conversation `visibility` is enforced on every read, not just 
          +in the UI:                                                    
      876 +                                                              
      877 +- **`shared`** — readable by any member of the workspace.     
      878 +- **`private`** — readable only where `author_id` = the caller
          +.                                                             
      879 +                                                              
      880 +The effective read predicate for conversations is therefore:  
      881 +`workspace_id = :wid AND (visibility = 'shared' OR author_id =
          + :uid)`. Private threads are                                  
      882 +invisible to everyone else by construction, independent of rol
          +e.                                                            
      883 +                                                              
      884 +## 11. Normalisation & Deliberate Denormalisation             
      885 +                                                              
      886 +The schema is **third normal form** — each non-key attribute d
          +epends on the whole key and                                   
      887 +nothing else (e.g. a member's role lives on `memberships`, nev
          +er duplicated onto messages;                                  
      888 +prompt tags are their own JSON domain, not repeated columns). 
          +Four denormalisations are                                     
      889 +kept on purpose, each a cached value with a clear maintainer: 
      890 +

      892 +|---|---|---|

          +ch every open | Set once at creation |

           +s a slow socket

           +briefly, then refetch

      1529 +  assignment is O(1) under a per-branch lock.                
      1530 +- **Why correct:** a single monotonic `seq` per branch gives

      1531 +  network arrival, so every client converges on the same conv

  Let me show you what's actually happening.

  What an LLM provider is

  Groq / Ollama / OpenAI are just HTTP APIs. You send them a list of messages
  (the conversation so far) and a model name; they stream the reply back to you 
  token by token. That streaming-text-from-a-model is the AI. Your code doesn't
  do the thinking — the model does. Your code just carries the tokens.

  What the conversation engine actually does (the whole thing)

  When someone hits Send:

  on send(conversation, user, text):
      save node(role=user, text)              # 1 DB write
      history = load nodes(conversation)      # the messages so far, in order
      stream  = provider.chat(history)        # 1 streaming HTTP call to
  Groq/Ollama
      reply = ""
      for token in stream:                    # tokens arrive from the model
          reply += token
          send token → browser (SSE)          # the user watches it type out
      save node(role=assistant, reply)        # 1 DB write

  That's it. The "engine" is a streaming relay + two database writes. The
  intelligence is entirely inside provider.chat(...) — one request, one streamed
  response. No loop, no branches, no state machine → nothing for LangGraph to 
  manage.

  And you already have the core of this — your Week-0 stub does precisely
  provider.stream(prompt) → SSE. The conversation engine just adds: send the
  full history (not one line), and persist the turns. That's the whole v1.

  Only when the AI has to do more than one step in a controlled loop — reason,
  then reflect on itself, then fan out to three reasoners, then decide whether
  to loop or stop. That's a graph of steps, and that's why Ouroboros uses "a
  whole lot of LangGraph."
