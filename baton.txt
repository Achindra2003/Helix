  §9 — Fork model (the clever/risky bit)

  A fork does not copy message history. It creates one new branch row whose
  fork_node_id points at the node you forked from. To read a branch's full
  history, you walk parent_id pointers from its head back up through the parent
  branch's nodes — shared structurally, like Git. So forking is O(1) regardless
  of conversation length. The one subtlety: a deep-reasoning branch also copies
  the engine's LangGraph checkpoint (so recursive state continues), but plain
  chat branches don't need that. Risk: the read-path (assembling history across
  branch boundaries) is where bugs hide, not the write. I think it's the right
  design — flagging so you know where to scrutinize later.

  §11 — Realtime split

  Sends/forks go through REST/SSE, not the WebSocket. The WS room is read-only:
  presence + broadcasting what already happened. Why: it gives one authoritative
  server order (the seq counter) instead of trying to merge concurrent socket
  writes (full operational-transform territory, which the plan explicitly scopes
  out). Simpler, and correct for "append-only shared log."

  Both are sound; nothing to change unless you disagree. Now building M1.

❯ explain what you've built and with what vision                                

● The vision (what we're serving)

  Helix is "Git for your team's AI work." The core insight from your own docs:
  people use AI alone in private tabs and lose track of what already worked.
  Helix makes AI a team resource — shared conversations, branchable threads, a
  reusable prompt library — and then adds a power tool on top: a monitored Deep 
  Reasoning mode for hard problems the whole team can watch, steer, and kill.

  Two design commitments shaped everything I built:

  - Core first, depth later. The collaborative workspace (M1–M5) is the product
  and must always be demoable. Deep Reasoning (M6–M7) is the headline but
  cuttable — never on the critical path. So I started at the foundation, not the
  flashy part.
  - Decisions and interfaces before infrastructure. You steered me here
  explicitly: lock the stack, write the contract, keep setup light. Build the
  skeleton you fully understand before standing up heavy machinery.

  What I built, and why each piece exists

  1. A locked tech stack + a written API contract (helix-api-contract.md).
  This is the load-bearing artifact. The contract is the interface both sides 
  build against in parallel — frontend and backend (or you across different work
  sessions) never block on each other because the wire format is agreed up
  front. It encodes the whole surface: REST for CRUD, WebSocket for
  presence/fan-out, SSE for token streaming, plus the RBAC matrix and a uniform
  error shape. Getting this right early is cheap; changing it after both sides
  build on it is expensive.

  2. A vertical slice (Week 0).
  /health (proves React→FastAPI→DB round-trip) and /chat/stream (proves a token
  streams end-to-end through a pluggable provider). The point wasn't features —
  it was proving the hardest path (live streaming across all three tiers) works
  on day one, so nothing downstream is a surprise.

  3. The pluggable LLM provider layer.
  Everything talks to an LLM only through one LLMProvider interface — stub,
  groq, ollama behind a single factory. This is the vision's model-agnostic
  promise made concrete: swap models by changing one env var, never code. The
  stub (echoes the prompt) means the whole app runs with zero keys, zero
  downloads — so the product is always demoable.

  4. M1 — the foundation every other module scopes to.
  Auth (JWT + bcrypt), workspaces, memberships with roles, invite links, and
  RBAC. This isn't glamorous, but it's the spine: every future resource
  (conversations, branches, prompts, runs) lives inside a workspace tenant and
  is gated by a role. I built the RBAC as a rank ladder (Owner > Collaborator > 
  Observer) via a require_role dependency — so "Observers can read but not steer
  a run" is one reusable check, not conditionals scattered everywhere. That
  directly serves the vision's governance story (the PM who can replay but not
  edit; the owner who controls who joins).

  5. Zero-infra dev on SQLite.
  When you said the heavy stuff was too early, I moved dev persistence to a
  SQLite file via SQLAlchemy's async layer. Same ORM code runs on Postgres in
  prod — you flip DATABASE_URL and nothing else changes. This keeps the "lock
  decisions, stay light" philosophy while keeping the production path honest (no
  SQLite-only shortcuts that'd break on Postgres later).

  The through-line

  Each thing earns its place by serving the next thing:

  contract  → so both sides build without blocking
    provider interface → so the product is model-agnostic AND always demoable
      M1 (workspaces + roles) → because every module scopes to a tenant + role
        SQLite-now / Postgres-later → light today, production-honest tomorrow
          ...M2 conversations → the node tree the fork engine (M4) will branch

  The deliberate ordering reflects the deepest decision: build the boring, 
  load-bearing core so well that the exciting part (Deep Reasoning) can land on 
  top without the foundation wobbling — and can be cut entirely if time runs 
  out, while still shipping a complete product. That's the whole risk posture of
  your 8-week plan, expressed in code.

  What I haven't built yet is everything that makes it feel alive —
  conversations, the branch tree, real-time presence, and the monitored
  reasoning mode. Those are next, and they all stand on the M1 spine that's now
  real and verified.

  the workspace starts to actually do something?
