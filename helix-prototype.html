The Problem

Teams use AI together — the tools assume they don't.
🔒

Siloed context

Everyone prompts in their own tab. Teammates can't see or continue each other's threads, so the same questions get re-asked.

→ redundant queries · wasted budget · lost progress
➡️

No branching

Linear chats can't fork. A promising thread can't be cloned to try an alternative without derailing the original.

→ can't explore two approaches in parallel
👁️

Unmonitored autonomy

Long, autonomous AI runs are opaque — no mid-run visibility, no stop control, no cost guardrail.

→ runaway compute before anyone can intervene

The Solution

One shared workspace + one power tool.
💬

Shared & private conversations

Every thread belongs to a workspace and is shared (team knowledge) or private. Nobody re-asks what a colleague already solved.

⑂

Fork & branch tree

Fork any conversation at any point into an independent branch that inherits full context — a Git-style lineage tree. O(1), no history copy.

📚

Shared prompt library

Prompts that worked are saved, tagged, and searchable — the team's accumulated record of what's been tried.

🟢

Real-time & presence

Live updates over WebSockets show who's in the workspace and stream responses as they generate. The workspace feels alive.

🧠

Deep Reasoning mode — the headline

Escalate a hard question into a recursive engine (reason → reflect → synthesize, looping to convergence). The whole team watches the live trace — with a kill switch, steer/pause, and a token/cost budget meter so a long run never becomes a runaway black box.

Scope discipline. The collaborative core (M1–M5, M8, M9) is the product and ships first on plain streaming chat. Deep Reasoning + monitor (M6–M7) is the headline depth feature and the cuttable layer — if engine integration runs long, Helix still ships and demos as a complete collaborative workspace.

How Helix is built

The same pieces the prototype simulates, mapped to the real system. Three transport channels, an append-only node tree that makes forking O(1), a role policy table, and a checkpointed reasoning engine.

Everything talks to an LLM only through one pluggable LLMProvider interface — swap models by changing an env var, never code. The engine is an in-process package the collaborative core does not depend on (so it stays cuttable).

2 Three transport channels — and why each

REST

Authoritative writes

Sends, forks, CRUD. One server assigns the canonical seq order — no concurrent-write merging.

WebSocket

Presence & fan-out

A per-workspace room broadcasts who's online and what already happened. Read-only — it never authors state.

SSE

Token streaming

LLM replies and the deep-reasoning trace stream token-by-token, rendering live as they generate.

3 The node tree — fork is O(1)

A fork does not copy history. It creates one branch row whose fork_node_id points at the node you forked from; reading a branch walks parent_id pointers back through the parent — shared structurally, like Git. Forking is O(1) regardless of length.

# a fork = one new row, no history copy class Branch: id # this branch conversation # tenant scope parent_branch # where it split from fork_node_id # the exact node it inherits up to # history(branch) = walk parent_id from head → root, crossing branch boundaries def history(branch): node = branch.head while node: yield node node = node.parent # O(depth), not O(copies)

4 Role-based access control (M9)

A single require_role rank ladder gates every action. Try the "view as" switch in the prototype — as an Observer, Send / Fork / Deep Reasoning / Steer / Kill all disable.

5 Deep Reasoning engine & the monitor (M6–M7)

Escalation runs a recursive LangGraph loop. Every node emits a step event over SSE; the monitor renders the topology, meters, and trace live. Guardrails make the autonomy safe: a token/cost budget, a request-rate limit, a recursion-depth guard, and an energy heuristic — any breach halts the run. State is checkpointed, so a killed run is resumable and forkable.
